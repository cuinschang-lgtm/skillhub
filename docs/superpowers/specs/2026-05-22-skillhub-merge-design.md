# SkillHub 双目录择优合并设计

## 背景

需要对比以下两份代码并合并为一份，同时保证网站基础功能可用：

- `/Users/cuins/Downloads/project_2026/skillhub`
- `/Users/cuins/Documents/Projects/project_2026/skillhub`

最终合并结果落在当前工作目录：

- `/Users/cuins/Documents/Projects/project_2026/skillhub`

本次目标不是做大规模重构，而是在尽量小的改动范围内，择优吸收两边差异，得到一份可继续开发和部署的统一代码。

## 目标

- 以当前工作目录作为唯一合并落点
- 对两份项目做源码级差异筛选，忽略生成物和缓存文件
- 优先保留能支撑网站基础功能的实现
- 合并后至少保证前后端基础链路完整
- 为后续继续开发保留清晰的文件边界和最小风险

## 非目标

- 不在本次合并中重写网站交互或视觉风格
- 不做与本次功能无关的大规模重构
- 不把两边所有差异机械叠加
- 不处理生成物、缓存文件、平台相关二进制差异

## 已确认事实

经目录和文件级对比，两个目录的主体结构高度一致，真正有意义的差异主要集中在以下区域：

- `apps/web/src/lib/api-proxy.ts`
- `render.yaml`
- `skeleton/*.py`

以下内容属于不应纳入合并决策的生成物或环境差异：

- `.next/`
- `node_modules/`
- `__pycache__/`
- `.DS_Store`
- `next-env.d.ts`
- 平台相关的 node 原生依赖产物

## 基线选择

以当前工作目录 `/Users/cuins/Documents/Projects/project_2026/skillhub` 作为合并基线。

选择理由：

- 当前目录已经包含更完整的前后端部署链路修复
- `apps/web/src/lib/api-proxy.ts` 支持 `SKILLHUB_API_BASE_URL` 和 `SKILLHUB_API_HOSTPORT` 两种后端定位方式，更适合本地和部署环境
- `render.yaml` 已具备前端服务、后端服务、数据库和关键环境变量配置，更接近可上线形态
- 当前目录中的 `skeleton/` 版本在正则表达式、类型标注和脚本导入路径上更一致，没有明显优于它的下载目录版本

## 合并策略

采用“当前目录为主，按文件择优吸收”的方式。

具体规则如下：

- 保留当前目录中已经验证更适合网站基础运行的文件
- 仅当下载目录中的某个源码文件明显补齐当前目录缺失能力时，才将其吸收
- 对差异较大的文件先做人工判断，再决定是否局部合并
- 不直接拷贝整个目录覆盖，避免把旧配置、缓存和错误改动带入当前基线

## 文件级决策

### 前端代理层

保留当前目录的 `apps/web/src/lib/api-proxy.ts`。

原因：

- 支持显式公网地址配置
- 支持内网 host:port 透传
- 保留生产环境未配置后端时的错误保护
- 更利于 Render 或类似平台进行前后端分离部署

### 部署配置

保留当前目录的 `render.yaml`。

原因：

- 同时声明了前端服务和后端服务
- 包含数据库配置
- 包含健康检查路径
- 包含前后端联动所需环境变量
- 更符合“网站基础功能可部署”的目标

### Python 骨架脚本

默认保留当前目录的 `skeleton/` 文件集，仅补入当前目录缺失但下载目录存在、且确实需要的源码文件。

当前判断：

- 下载目录中的 `skeleton/__init__.py` 可能是补充项，但不属于网站基础功能核心阻塞项
- 其他差异主要表现为旧式类型标注、错误的正则替换痕迹或较弱的实现表达，不应反向覆盖当前目录

### 其余网站相关目录

`apps/api/` 与 `apps/web/` 的页面、路由和接口主体结构基本一致，当前阶段不做大范围互相覆盖，仅在实际运行验证暴露问题时做针对性修补。

## 基础功能定义

本次“网站基础功能可用”按以下范围定义：

- 前端首页、登录页、注册页可访问
- 工作台、上传、知识库、进度、benchmark、设置页可访问
- Next 侧 `/api/*` 路由能正常代理到后端
- FastAPI 基础接口可启动并返回数据
- `/health` 健康检查可用
- 前端可以读取任务列表与技能列表

## 执行步骤

1. 以当前工作目录为基线再次确认差异文件
2. 清除合并范围内的生成物判断干扰，只看源码
3. 逐项合并关键文件，优先处理前端代理和部署配置
4. 检查 `skeleton/` 是否存在必须从下载目录补入的遗漏文件
5. 运行基础验证，确认前后端关键链路可用
6. 如果出现问题，仅做针对当前阻塞点的最小修复

## 错误处理原则

- 若发现下载目录中的改动只是生成物差异，直接忽略
- 若发现当前目录存在未跟踪文件且与本次修改区域重叠，先读取并确认后再继续
- 若运行验证暴露前后端地址、跨域、数据库连接或导入路径问题，优先修复这些基础阻塞问题
- 若某项差异无法在短时间内证明收益，则优先保守处理，避免引入回归

## 测试与验证

实现阶段完成后，至少进行以下验证：

- Python 侧 `skeleton/*.py` 语法检查
- 前端依赖和构建检查
- 后端服务启动检查
- 前端关键页面访问检查
- 前端经由 `/api/*` 调用后端的关键链路检查

若时间和环境允许，可补充：

- 本地前后端联调
- 上传任务从创建到查询状态的最小闭环验证

## 风险与缓解

风险：

- 当前目录存在未跟踪文件，后续编辑时若与本次改动重叠，可能带来不确定性
- 部署配置中的域名和数据库资源依赖外部平台，真实部署时仍需按实际环境调整
- `skeleton/` 的执行链路可能受运行环境、外部 API key 和 PDF 处理依赖影响

缓解：

- 本次仅修改明确纳入合并范围的文件
- 保持文件级最小改动
- 先验证基础功能，再决定是否扩大修复范围

## 产出

本次设计对应的预期产出为：

- 一份统一后的 `SkillHub` 代码目录
- 保留当前目录中更适合网站基础运行的关键实现
- 一组最小验证结果，用于确认基础网站能力可用
